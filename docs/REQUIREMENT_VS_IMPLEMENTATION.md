# 智枢AI SaaS系统 — 需求 vs 实现对照评估报告

> 评估日期：2026-06-17  
> 对照文档：`智枢 AI SaaS 系统开发需求.txt`  
> 评估基准：能否让客户真实投入使用的生产级别

---

## 总体结论

**当前系统距离"能让客户真实投入使用"还有较大差距。** 15项核心功能核查中，仅4项真正可用，其余要么是空壳（PLACEHOLDER），要么完全未实现。核心业务流程（自媒体发布到平台、招聘平台对接、获客自动引流、定时发布）都停留在数据库CRUD层面，没有对接任何第三方平台。APK端更有8个核心屏幕使用硬编码假数据。

**预估投入生产所需开发量：约 30-40 人天**

---

## 一、需求逐项对照

### 1. 自媒体版块（⭐⭐⭐⭐⭐ 核心必开发）

| 需求项 | 需求描述 | 实现状态 | 差距说明 |
|--------|---------|---------|---------|
| 1.1 AI智能内容生成 | 标题/话题/文案/图生文/小红书图文/图片/短视频/数字人短视频 | **部分实现** | 文本生成可用，但图片/视频/数字人生成只是返回模板文本，未对接真实生成API（腾讯云TokenHub视频生成、图像生成等） |
| 批量剪辑 | 短视频批量剪辑 | **未实现** | 无任何批量剪辑代码 |
| 批量生成 | 支持选择生成数量 | **部分实现** | AI文本生成有数量参数，但图片/视频不支持 |
| 图生文 | 图片转文字描述 | **未实现** | 无多模态模型调用 |
| 图生视频 | 图片生成宣传视频 | **未实现** | 无视频生成API对接 |
| 1.2 矩阵系统 | 多账号管理+扫码授权 | **严重缺失** | 后端有Playwright扫码逻辑但极不稳定；APK端MatrixAccountScreen使用Mock数据，`loadAccounts`被注释掉；小红书标记"即将上线"，视频号完全没配置；添加账号是手动填写，不是扫码 |
| 统一管理 | 跨平台账号统一管理 | **部分实现** | 数据模型完整，APK端展示全假数据 |
| 1.3 批量发布 | 一键上传到各平台 | **占位符** | `content-publish.ts`的`POST /publish`只写数据库记录；`batch-publish`直接硬编码返回`{success: true}`；**没有对接任何平台API**（抖音开放平台、快手开放平台等） |
| 自动添加标题标签 | 根据AI生成结果自动填标题/标签 | **未实现** | 发布流程无此逻辑 |
| 定时发布 | 根据平台规则设定发布时间 | **未实现** | 有`scheduledAt`字段和`scheduled`状态，但无定时执行引擎（无cron/bull），任务永远不会执行 |

| 下载生成内容 | 下载图片/短视频/数字人视频 | **部分实现** | 有文件URL字段，但生成的是假内容 |
| 1.4 数据统计 | 发布记录/播放量/点赞/评论/分享/粉丝增长 | **占位符** | `report.ts.disabled`被禁用；`statistics.ts.disabled`被禁用；APK端DashboardScreen全部硬编码假数据 |

**自媒体版块结论：核心的"发布到平台"链条完全断裂。AI生成了内容→保存到素材库→但无法发布到任何平台。矩阵扫码授权不稳定。这是最大的缺口。**

## 三、能否让客户使用？— 分场景判断

### 场景A：仅使用AI对话+文本生成 → ✅ 可以
AI对话、文案生成、标题/话题生成可用，内容自动保存到素材库。但无法发布到任何平台，只能手动复制使用。

### 场景B：使用自媒体运营全流程 → ❌ 不可以
AI生成→素材库这一步可用，但**素材无法发布到任何平台**，矩阵扫码授权不稳定，数据统计不可用。整个自媒体运营链条在"发布"环节断裂。

### 场景C：使用招聘全流程 → ❌ 不可以
只有本地职位管理，无法发布到招聘平台，无AI简历筛选，无自动回复，无面试管理。

### 场景D：使用获客全流程 → ❌ 不可以
天眼查/高德采集可用，但核心的"自动发送引流消息"未实现，二维码追踪未实现。

### 场景E：使用推荐分享 → ⚠️ 勉强可用
数据模型完整，但无实际二维码图片生成，无扫码落地页，追踪数据不会自动更新。

## 五、建议分步上线策略

### 第一步：MVP上线（约7天，完成U1-U7 + C1 + C2 + C4）

让客户可以：
- 正常登录使用（无安全漏洞）
- AI生成文本/图片/视频（真实内容）
- 内容自动保存到素材库
- 素材发布后标记为已使用

## 六、关于第三方平台对接的特别说明

需求文档中多次提到"网页自动化"方案来实现平台对接。这是当前最大的技术风险：

1. **抖音/快手/小红书/视频号**：这些平台没有公开的内容发布API。网页自动化（Playwright/Puppeteer）方案存在以下风险：
   - 平台反爬虫检测，频繁登录可能封号
   - 登录态（Cookie）有效期有限，需频繁重新授权
   - 平台UI变更会导致自动化脚本失效
   - 合规风险（平台用户协议通常禁止自动化操作）

2. **BOSS直聘/前程无忧/智联招聘**：同样没有公开API。网页自动化风险同上。

3. **建议方案**：
   - 短期：采用"半自动"模式——系统生成内容+复制到剪贴板，用户手动到各平台APP发布。降低技术风险和合规风险。
   - 中期：对接各平台开放平台API（如有），或使用平台官方的创作者服务中心。
   - 长期：如果必须全自动化，需要投入持续维护成本来应对平台反爬更新。

## 七、关键决策点

在开始开发之前，需要确认以下决策：

1. **自媒体发布方案选择**：全自动（网页自动化，高风险高成本）vs 半自动（生成内容+引导手动发布，低风险低成本）？
2. **招聘平台对接深度**：只做本地JD管理+AI筛选（低成本）vs 全自动发布到招聘平台（高成本高风险）？
3. **获客引流方案**：自动发送引流消息（有被封号风险）vs 仅采集潜客信息+手动联系？
4. **图像/视频生成API**：确认使用腾讯云TokenHub还是阿里云百炼？API Key和额度是否已申请？
5. **短信服务商**：验证码短信需要配置真实短信通道（阿里云/腾讯云短信），.env中只配置了DASHSCOPE，未配置短信服务。
6. **上线时间要求**：是否有客户等待上线的时间压力？这决定了优先做哪些功能。
